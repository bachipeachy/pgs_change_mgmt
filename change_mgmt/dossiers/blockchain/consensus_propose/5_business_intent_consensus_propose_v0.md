## Section 1 — Subdomain Purpose

**What this CR does:**

- Runs a fixed list of consensus slots from start to finish.
- Each slot: submit typed transactions to the mempool, then propose a block.
- After all slots: verify that blocks were proposed and all transaction types are present.
- This is the first time actors, wallets, validators, transactions, and block proposal all run together end-to-end.

**What it fixes along the way:**

- Block proposal was missing slot and epoch. Fixed.
- Five SYSTEM transaction workflows (MINT, BURN, POOL, REWARD, SLASH) were not wired to the deduplication step. Fixed.
- Validator registration stored different field names than the eligibility query was reading. Fixed.

**What it does NOT do:**

- No attestation. No finalization. No chain commitment.
- The block stops at PROPOSED state. The rest is deferred to sister CRs.

## Section 3 — Business Object Semantics

| Store Name | Business Record Model | Business Rationale |
|------------|-----------------------|-------------------|
| BLOCKS | Mutable State | One record per block_id; carries current block status (PROPOSED in V0); must be queryable by slot and proposer; correction must be possible before attestation |
| CONSENSUS_ROUNDS | Append-Only Journal | Immutable record of each slot execution outcome — proposer selected or round skipped; deletion would break consensus audit trail |
| TRANSACTION (mempool) | Mutable State | One record per tx_id; PENDING at mempool entry; status advances in later CRs; content-addressed by tx_id so duplicate admission is detectable |
| VALIDATOR | Mutable State | Authoritative validator registry; one record per actor; eligibility is read from this store at every proposer selection; fields must be canonical and consistent across write and read paths |

## Section 4 — Identity Semantics

### Primary Identity — Block

**Field:** `block_id`
**Source:** Generated by `CC_FORM_BLOCK_V0` at block formation; content-addressed from proposer, slot, epoch, and tx payload
**Role:** Unique handle for a proposed block; referenced by the consensus round record to link outcome to artifact
**Uniqueness:** One block per slot — each slot produces at most one block_id

### Primary Identity — Consensus Round

**Field:** `round_id`
**Source:** Supplied by caller as part of the slot descriptor (equals slot number in this CR)
**Role:** Identifies which slot this round record describes; links to block_id if a block was proposed, null if skipped
**Uniqueness:** One round record per slot; append-only — no update after write

### Primary Identity — Transaction

**Field:** `tx_id`
**Source:** Content-addressed from payload via hash derivation — same payload always produces the same tx_id
**Role:** Deduplication key; mempool admission is gated on tx_id uniqueness; blocks carry tx_id lists
**Uniqueness:** One mempool record per tx_id; second submission of identical payload is rejected as duplicate

### Primary Identity — Validator

**Field:** `validator_id`
**Source:** Generated at validator registration; deterministic from actor_id and registration payload
**Role:** Cross-subdomain handle; proposer selection records validator_id in block and round records
**Uniqueness:** One validator record per actor_id; duplicate registration is rejected by existence check

### Cross-Subdomain Identity Usage

actor_id is the shared identity anchor between the identity subdomain and the validator subdomain. A validator record cannot exist without a corresponding actor record — actor existence is enforced as a hard prerequisite before validator write. The consensus loop inherits validator_id from the validator registry at proposer selection; it does not re-validate actor existence.

## Section 5 — Business Invariants

| # | Invariant | Business Reason |
|---|-----------|-----------------|
| 1 | A block reaches PROPOSED state only in this CR — no attestation, finalization, or chain commitment | Scope boundary: the consensus lifecycle beyond proposal is deferred to sister CRs; premature state advancement would produce invalid chain state |
| 2 | Termination of the consensus loop is declared in the input (finite slot_schedule) — not imposed by an external kill signal or runtime flag | PGS zero-inference doctrine: no ambient control; the bound must be a governance contract, not a side-channel dependency |
| 3 | slot = round_number % 32; epoch = round_number // 32 — computed by the caller before invoking block proposal | Slot arithmetic is ETH convention; delegating it to the caller keeps the single-slot WF lean and avoids introducing a computation step inside the governed proposer selection path |
| 4 | Validator eligibility requires status = ACTIVE_ONGOING AND effective_balance present | Only fully activated validators with declared stake participate in proposer selection; inactive or stake-less records must never influence proposer rotation |
| 5 | SYSTEM authority transactions (MINT, BURN, POOL, REWARD, SLASH) require no actor wallet — admission is gated on triggered_by authority only | These are protocol-level supply operations; requiring a wallet would impose an ENDUSER constraint on a SYSTEM actor, breaking the authority model |
| 6 | ENDUSER authority transactions (TRANSFER, STAKE, UNSTAKE) require actor existence and a wallet | Value transfer requires an identifiable, provisioned participant; walletless actors cannot move funds |
| 7 | tx_id is content-addressed — same payload always produces the same tx_id; no nonce reservation | Determinism invariant: identical inputs must produce identical outputs across any number of executions; nonce reservation is stateful and non-deterministic |
| 8 | All CS writes are scoped to the owning subdomain's stores — no cross-subdomain writes inside any capability contract | PGS isolation constraint; CS scope is a governance boundary, not a runtime option |
| 9 | A consensus round record is written regardless of whether a block was proposed or skipped | The audit trail must be complete — a missing round record is indistinguishable from a runtime failure; skipped rounds are governance-visible outcomes |

## Section 9 — Workflows (WF)

### WF_RUN_CONSENSUS_SLOTS_V0
**Summary:** Executes a finite, declared sequence of consensus slots — submitting typed transactions to the mempool and proposing a block per slot — then verifies the results against declared expectations.
**Subdomain:** `consensus_pos`
**Start Node:** `IN_CONSENSUS_SLOTS_V0`

**Execution Nodes:**

| Node | Type | Routing Outcomes |
|------|------|-----------------|
| IN_CONSENSUS_SLOTS_V0 | IN | ACK → CC_EXECUTE_SLOT_SEQUENCE_V0, NACK → EXIT |
| CC_EXECUTE_SLOT_SEQUENCE_V0 | CC | SUCCESS → CC_VERIFY_SLOT_RESULTS_V0, VIOLATION → EXIT |
| CC_VERIFY_SLOT_RESULTS_V0 | CC | SUCCESS → EXIT_SLOTS_COMPLETE, VIOLATION → EXIT |
| EXIT_SLOTS_COMPLETE | EXIT | — |
| EXIT | EXIT | — |

**CC Dependencies:** `CC_EXECUTE_SLOT_SEQUENCE_V0`, `CC_VERIFY_SLOT_RESULTS_V0`

*Note: `CC_EXECUTE_SLOT_SEQUENCE_V0` absorbs the N-slot iteration internally using the Collatz pattern — the WF DAG is linear; the loop lives inside the CC. Per slot, the CC internally calls `WF_PROPOSE_BLOCK_V0` and the eight typed transaction WFs; these are governed invocations, not direct CC-to-CC calls.*

## Section 10 — Capability Contracts (CC)

### CC_EXECUTE_SLOT_SEQUENCE_V0 (NEW)
**Summary:** Absorbs the N-slot iteration of the consensus loop. For each slot in the declared schedule: submits the slot's typed transactions to the mempool via the appropriate typed WFs, then invokes `WF_PROPOSE_BLOCK_V0` for block proposal. Exits when the schedule is exhausted. This is the Collatz pattern applied to side-effectful iteration — the WF DAG stays linear; all looping is inside this CC.

**Inputs:**

| Field | Type | Required |
|-------|------|----------|
| slot_schedule | array | YES |
| triggered_by | string | YES |

**Outputs:**

| Field | Type |
|-------|------|
| slots_executed | integer |
| blocks_proposed | integer |
| rounds_skipped | integer |
| tx_submitted | integer |

**Result Statuses:** `SUCCESS`, `VIOLATION`

**Pipeline Steps:**

| Step | Purpose |
|------|---------|
| iterate_slot_schedule | Walk the ordered slot list from start to end; for each slot, execute the transaction submission and block proposal steps |
| submit_slot_transactions | For each transaction descriptor in the slot's transaction list, invoke the matching typed transaction WF (TRANSFER, STAKE, UNSTAKE, MINT, BURN, POOL, REWARD, SLASH) to persist the tx to the mempool |
| invoke_block_proposal | Invoke `WF_PROPOSE_BLOCK_V0` with round_number, slot, epoch, timestamp, and triggered_by for this slot; outcome (proposed or skipped) is recorded by the WF |
| accumulate_counters | Track slots executed, blocks proposed, rounds skipped, and txs submitted for the result payload |
| exit_on_schedule_exhausted | When the slot list is fully consumed, return SUCCESS with counters; no polling, no external signal |

---

### CC_VERIFY_SLOT_RESULTS_V0 (NEW)
**Summary:** Post-run assertion over the BLOCKS and TRANSACTION stores. Confirms that at least one block was proposed in PROPOSED state during the run, and that all eight transaction types are represented in the mempool in PENDING state. Produces a VIOLATION if either assertion fails.

**Inputs:**

| Field | Type | Required |
|-------|------|----------|
| expected_block_count | integer | YES |
| expected_tx_types | array | YES |

**Outputs:**

| Field | Type |
|-------|------|
| blocks_found | integer |
| tx_types_found | array |
| violations | array |

**Result Statuses:** `SUCCESS`, `VIOLATION`

**Pipeline Steps:**

| Step | Purpose |
|------|---------|
| read_block_store | Read all records from the BLOCKS store and filter for status = PROPOSED; count qualifying records |
| assert_block_count | Verify at least one PROPOSED block exists; if none found, record a VIOLATION for "no blocks proposed" |
| read_transaction_store | Read all records from the TRANSACTION store and filter for status = PENDING |
| assert_tx_type_coverage | Verify all eight expected tx_types appear in the PENDING set; record a VIOLATION for each missing type |
| return_result | Return SUCCESS if all assertions pass; return VIOLATION with the violations list if any assertion failed |

---

### CC_QUERY_ELIGIBLE_VALIDATORS_V0 (UPDATED)
**Summary:** Reads the VALIDATOR store and returns the set of validators eligible for proposer selection. Eligibility requires status = ACTIVE_ONGOING and effective_balance present. The filter fields are updated in this CR to match the canonical fields written by `CC_WRITE_VALIDATOR_RECORD_V0`; the retired fields (enrollment_status, stake) are removed from the filter.

**Inputs:**

| Field | Type | Required |
|-------|------|----------|
| epoch | integer | NO |

**Outputs:**

| Field | Type |
|-------|------|
| eligible_validators | array |
| count | integer |

**Result Statuses:** `SUCCESS`, `NOT_FOUND`, `VIOLATION`, `BACKEND_ERROR`

**Pipeline Steps:**

| Step | Purpose |
|------|---------|
| read_validator_store | Read all records from the VALIDATOR store |
| filter_eligible | Apply eligibility filter: status must equal ACTIVE_ONGOING AND effective_balance must be present and non-null; discard all records that fail either condition |
| return_eligible_set | Return the qualifying set with count; if the set is empty, return NOT_FOUND — this signals the calling WF to skip the round rather than halt |
