# Business Intent: blockchain / mempool
**Domain:** blockchain
**Subdomain:** mempool
**Version:** V0
**Status:** APPROVED

---

## Section 1 — Subdomain Purpose

The mempool subdomain governs the pending transaction buffer within the blockchain domain. It is the governed staging area between transaction submission and block commitment. A transaction enters the mempool when it has been validated and signed by the submitter; it leaves when a block proposer drains it during block formation. The mempool subdomain owns this lifecycle entirely — it does not share custody of pending transactions with the transaction subdomain (which governs submission) or the consensus subdomain (which governs proposer selection and round recording). The mempool exists because pending and committed transactions are architecturally distinct: pending transactions are ephemeral, mutable, and drainable; committed transaction records are durable and lifecycle-tracked. Without a governed mempool boundary, these concerns collapse into one store, producing an ungoverned staging state with no defined drain mechanism.

## Section 2 — Scope Boundary

### In Scope — V0

| Capability | Notes |
|-----------|-------|
| Stage pending transaction to MEMPOOL store | Deduplication enforced; arrival timestamp recorded |
| Query pending transactions for block proposal | Returns all pending entries in arrival order; no filtering by actor or type |
| Drain consumed transactions from MEMPOOL store | DELETE-only; called as part of block formation; no write to TRANSACTION store |

### Explicitly Deferred — Not In Scope

| Capability | Deferred To |
|-----------|-------------|
| TRANSACTION store status update (PENDING → COMMITTED, block association) | Future CR — revisit JIT |
| Nonce management and per-actor transaction sequencing | Future CR |
| Mempool expiry and eviction of stale transactions | Future CR |
| Priority ordering and fee-based transaction selection | Future CR |
| Multi-node mempool propagation | Future CR — requires P2P transport |

## Section 3 — Business Object Semantics

| Store Name | Business Record Model | Business Rationale |
|------------|----------------------|-------------------|
| MEMPOOL | Mutable State | Pending transactions are ephemeral entries — written on submission, deleted on drain. Corrections are possible before drain. Deletion is the normal lifecycle exit, not an exception. Mutable state is the correct model for a buffer that must support both write and delete. |

---

## Section 4 — Identity Semantics

### Primary Identity

**Field:** `tx_id`
**Source:** Generated during transaction submission (transaction subdomain pipeline)
**Role:** Unique key for each pending transaction in the MEMPOOL store; used as the drain key when the proposer removes consumed transactions
**Uniqueness:** One mempool entry per tx_id; duplicate submission is rejected by deduplication check before write

### Secondary Identity

**Field:** `tx_hash`
**Role:** Cryptographic fingerprint of the signed transaction bytes; prevents the same signed transaction from being submitted twice under a different tx_id
**Uniqueness:** Must be unique across all mempool entries; enforced by hash deduplication check

### Cross-Subdomain Identity Usage

`tx_id` is generated by the transaction submission pipeline (transaction subdomain) and passed into the mempool CC as an input. The mempool does not generate tx_ids — it receives them. `actor_id` and `wallet_id` are carried as metadata on the mempool record for observability but are not used as lookup keys within this subdomain.

## Section 5 — Business Invariants

| # | Invariant | Business Reason |
|---|-----------|-----------------|
| 1 | A pending transaction must not be written to the MEMPOOL store if a record with the same tx_id already exists | Duplicate transactions would cause the same economic action to be executed twice; deduplication is a hard prerequisite |
| 2 | A pending transaction must not be written to the MEMPOOL store if a record with the same tx_hash already exists | The same signed transaction submitted under a different tx_id must be rejected; hash uniqueness is the cryptographic deduplication guard |
| 3 | Mempool entries are ordered by arrival time; the proposer must consume them in that order | Sequential ordering is the V0 fairness model; reordering is not permitted without a future fee market CR |
| 4 | Drain must not write to the TRANSACTION store | The TRANSACTION store is owned by the transaction subdomain; mempool drain is DELETE-only from the MEMPOOL store |
| 5 | The MEMPOOL store and TRANSACTION store are independent stores; no record spans both | Subdomain isolation — each subdomain owns its own storage path; cross-store records would violate the isolation invariant |

## Section 7 — Actors

This subdomain does not define new actors. Actor authority classes are defined in `5_business_intent_identity_v0.md` (AC_ENDUSER_V0, AC_SYSTEM_V0). The Transaction Submitter acts under enduser authority; the Block Proposer acts under system authority within the consensus round execution.

## Section 8 — Intents (IN)

This subdomain does not define standalone Intents. The mempool subdomain is a governed capability provider — it owns three Capability Contracts invoked by workflows in other subdomains. Admission for each action is governed by the calling workflow's Intent:

- WRITE_TO_MEMPOOL is invoked from the transaction submission workflow, admitted by `IN_TRANSACTION_SUBMITTED_V0`
- QUERY_MEMPOOL and DRAIN_MEMPOOL are invoked from the block proposal workflow, admitted by `IN_BLOCK_PROPOSED_V0`

No standalone `IN_` artifacts are declared by this subdomain in V0.

---

## Section 9 — Workflows (WF)

This subdomain does not declare standalone Workflows. The mempool CCs are invoked as governed steps within workflows owned by peer subdomains.

**WRITE_TO_MEMPOOL** is called from `WF_SUBMIT_TRANSACTION_V0` (owned by `blockchain::transaction`) after transaction hashing, replacing the current step that writes to the TRANSACTION store.

**QUERY_MEMPOOL and DRAIN_MEMPOOL** are called from `WF_PROPOSE_BLOCK_V0` (owned by `blockchain::consensus_pos`). QUERY_MEMPOOL is called before block formation; DRAIN_MEMPOOL is called after block formation.

Updated `WF_PROPOSE_BLOCK_V0` node sequence:

| Node | Type | Routing Outcomes |
|------|------|-----------------|
| IN_BLOCK_PROPOSED_V0 | IN | ACK → CC_QUERY_ELIGIBLE_VALIDATORS_V0, NACK → EXIT |
| CC_QUERY_ELIGIBLE_VALIDATORS_V0 | CC | SUCCESS → CC_SELECT_PROPOSER_V0, VIOLATION → EXIT, BACKEND_ERROR → EXIT |
| CC_SELECT_PROPOSER_V0 | CC | SUCCESS → CC_QUERY_MEMPOOL_TXS_V0, VIOLATION → EXIT |
| CC_QUERY_MEMPOOL_TXS_V0 | CC | SUCCESS → CC_FORM_BLOCK_V0, VIOLATION → CC_SKIP_ROUND_V0, BACKEND_ERROR → EXIT |
| CC_FORM_BLOCK_V0 | CC | SUCCESS → CC_DRAIN_MEMPOOL_V0, VIOLATION → EXIT, BACKEND_ERROR → EXIT |
| CC_DRAIN_MEMPOOL_V0 | CC | SUCCESS → CC_RECORD_CONSENSUS_ROUND_V0, VIOLATION → EXIT, BACKEND_ERROR → EXIT |
| CC_RECORD_CONSENSUS_ROUND_V0 | CC | SUCCESS → EXIT_SUCCESS, VIOLATION → EXIT, BACKEND_ERROR → EXIT |
| CC_SKIP_ROUND_V0 | CC | SUCCESS → EXIT, VIOLATION → EXIT, BACKEND_ERROR → EXIT |
| EXIT_SUCCESS | EXIT | — |
| EXIT | EXIT | — |

## Section 10 — Capability Contracts (CC)

### CC_WRITE_MEMPOOL_TX_V0
**Summary:** Stage a validated, signed transaction into the MEMPOOL store. Enforce deduplication by tx_id and tx_hash before write. Record arrival timestamp for ordering.

**Inputs:**

| Field | Type | Required |
|-------|------|----------|
| tx_id | string | YES |
| tx_hash | string | YES |
| tx_type | string | YES |
| actor_id | string | NO |
| from_wallet_id | string | NO |
| to_wallet_id | string | NO |
| amount | number | YES |
| created_at | string (datetime) | YES |

**Outputs:**

| Field | Type |
|-------|------|
| result_status | string |

**Result Statuses:** `SUCCESS`, `ALREADY_EXISTS`, `VIOLATION`, `BACKEND_ERROR`

**Pipeline Steps:**

| Step | Purpose |
|------|---------|
| generate_mempool_key | Generate a stable lookup key for this entry in the MEMPOOL store |
| check_tx_id_duplicate | Verify no entry with this tx_id exists in the MEMPOOL deduplication index; reject if duplicate to prevent double-staging |
| write_pending_tx | Write the pending transaction record to the MEMPOOL store keyed by tx_id |
| register_tx_hash | Register the tx_hash in the deduplication index; reject if the same signed transaction was already staged under a different tx_id |

---

### CC_QUERY_MEMPOOL_TXS_V0
**Summary:** Read all pending transactions from the MEMPOOL store and return them in arrival order for block formation. Returns VIOLATION if the mempool is empty — an empty mempool means no block can be formed and the round should be skipped.

**Inputs:**

_No inputs declared._

**Outputs:**

| Field | Type |
|-------|------|
| tx_ids | array of string |

**Result Statuses:** `SUCCESS`, `VIOLATION`, `BACKEND_ERROR`

**Pipeline Steps:**

| Step | Purpose |
|------|---------|
| list_mempool_entries | Read all records from the MEMPOOL store |
| assert_mempool_non_empty | Confirm at least one pending transaction exists; return VIOLATION if empty so the calling workflow can route to skip-round |

---

### CC_DRAIN_MEMPOOL_V0
**Summary:** Remove consumed transactions from the MEMPOOL store after a block has been formed. Accepts the list of tx_ids included in the block and deletes each from the MEMPOOL store. Idempotent — a NOT_FOUND on any tx_id is not an error.

**Inputs:**

| Field | Type | Required |
|-------|------|----------|
| tx_ids | array of string | YES |
| block_id | string | YES |

**Outputs:**

| Field | Type |
|-------|------|
| drained_count | integer |

**Result Statuses:** `SUCCESS`, `VIOLATION`, `BACKEND_ERROR`

**Pipeline Steps:**

| Step | Purpose |
|------|---------|
| delete_consumed_txs | Delete each tx_id in the input list from the MEMPOOL store; idempotent — NOT_FOUND is treated as already drained |
