# Business Intent: blockchain / chain
**Domain:** blockchain
**Subdomain:** chain
**Version:** V0
**Status:** COMPLETE

---

## Section 1 — Subdomain Purpose

The `chain` subdomain governs the canonical, immutable sequence of committed blocks within the blockchain domain. It establishes the authoritative history of all transactions and state changes by maintaining an ordered record of block references. This subdomain ensures the integrity and continuity of the blockchain from its genesis, providing a verifiable and unalterable ledger. It owns chain progression — block content formation and proposal remain owned by their existing subdomains.

## Section 3 — Business Object Semantics

| Store Name | Business Record Model | Business Rationale |
| --- | --- | --- |
| CHAIN | Append-Only Journal | The chain is the immutable, ordered history of the blockchain. Once a block reference is committed it can never be altered or deleted; audit integrity and historical accuracy depend on it. |
| CHAIN_STATE | Mutable State | Tracks the current head and height as one record, updated atomically with each commitment. It is the canonical head authority — head state is never derived by traversing CHAIN. |
| CHAIN_EVENTS | Append-Only Journal | Permanent record of chain lifecycle facts (genesis committed, block committed); deletion would break audit integrity. |

## Section 4 — Identity Semantics

### Primary Identity (for CHAIN entries)

**Field:** `block_id`
**Source:** Generated by `CC_FORM_BLOCK_V0` during block proposal (genesis: by the block-owned genesis formation capability).
**Role:** Unique identifier for a block; primary key for referencing block content in the `BLOCKS` store and anchoring chain entries.
**Uniqueness:** One unique `block_id` per block; enforced by the block formation process.

### Secondary Identity (for CHAIN entries)

**Field:** `block_hash`
**Role:** Cryptographic hash of the block's content, computed at commitment time (proposed blocks carry no hashes). Provides the linkage key: each entry's `previous_block_hash` equals the prior head's `block_hash`.
**Uniqueness:** Unique per block content; collisions practically negligible.

### Primary Identity (for CHAIN_STATE)

**Field:** fixed key (single canonical record)
**Source:** System-defined.
**Role:** Identifies the one canonical chain state record.
**Uniqueness:** Exactly one CHAIN_STATE record per blockchain instance.

### Cross-Subdomain Identity Usage

`block_id` is the identity anchor shared with `blockchain::block`, where full block content resides; the chain uses it to read proposed blocks and to request canonical marking. `genesis.actor` is registered in `blockchain::identity`; its MINT wallet resides in `blockchain::wallet`; the genesis validator is registered in `blockchain::consensus_pos`.

## Section 5 — Business Invariants

| # | Invariant | Business Reason |
|---|-----------|-----------------|
| 1 | The chain is an immutable, ordered sequence of block references. | Integrity and historical accuracy of the ledger; prevents tampering. |
| 2 | The blockchain is a closed monetary system; money neither leaves nor enters. | Accounting reconciliation; prevents external inflation/deflation. |
| 3 | Genesis funds the chain with 1,000,000 BachiCoin in a "MINT" wallet owned by `genesis.actor`. | Establishes the fixed initial supply of the closed system. |
| 4 | **Canonical Chain Progression** — A block may be committed only if it exists with status PROPOSED and its `round_id` equals current head height + 1; its `previous_block_hash` is linked to the current head hash at commitment. | Guarantees structural integrity: each block correctly follows its predecessor in order. |
| 5 | `BLOCKS` holds full block content; the chain stores references only. | Separation of concerns: the chain is a lightweight immutable index; content is managed by its owning subdomain. |
| 6 | **Genesis Execution** — Chain initialization may execute exactly once. After initialization, chain governance owns progression, not creation. | Prevents confusion between bootstrap and runtime behavior; single controlled initialization. |
| 7 | **Single Canonical Chain Model** — V0 supports a single canonical chain; fork selection, reorganization, attestation, finalization are out of scope. | Bounds the V0 model; prevents assumptions about advanced features. |
| 8 | Chain capabilities write only chain-owned stores; every `BLOCKS` write is performed by a block-owned capability. | Cross-subdomain writes are forbidden; store ownership is a governance boundary. |

## Section 8 — Intents (IN)

### IN_INITIALIZE_CHAIN_V0
**Summary:** Admits the one-time request to initialize the blockchain with a genesis block and initial funding.
**Workflow Binding:** `WF_INITIALIZE_CHAIN_V0`

**Input Fields:**

| Field | Type | Required | Description |
| --- | --- | --- | --- |
| initial_state | object | YES | Initial configuration embedded in the genesis block — the auditable starting condition of the ledger. |
| genesis_actor_id | string | YES | Identity of the privileged funding actor; enables actor registration and wallet ownership binding. |
| mint_wallet_id | string | YES | The MINT wallet to be created and funded; anchors the closed system's supply. |
| initial_supply | integer | YES | Amount minted at genesis (1,000,000 BachiCoin); fixes the total supply invariant. |
| validator_actor_id | string | YES | The genesis validator to register, enabling the consensus loop to produce blocks. |
| timestamp | string | YES | ISO 8601 genesis creation time; recorded in the genesis block for determinism and audit. |

**Outcomes:**

| Outcome | Description |
| --- | --- |
| ACK | Initialization request accepted for processing. |
| NACK | Rejected — invalid payload. (An already-initialized chain is detected by the first workflow step, not by the intent.) |

### IN_COMMIT_BLOCK_V0
**Summary:** Admits a request to commit a proposed block's reference to the canonical chain.
**Workflow Binding:** `WF_COMMIT_BLOCK_V0`

**Input Fields:**

| Field | Type | Required | Description |
| --- | --- | --- | --- |
| block_id | string | YES | Identifies the proposed block to commit; used to read its record from `BLOCKS`. |
| round_id | integer | YES | Declared height of the block; checked for sequential correctness against chain state. |
| triggered_by | string | YES | Actor context of the invoking slot workflow; preserved for audit. |
| timestamp | string | YES | ISO 8601 commitment time; recorded in the chain entry and lifecycle event. |

*`block_hash` and `previous_block_hash` are NOT inputs — proposed blocks carry no hashes; both are computed/linked during commitment.*

**Outcomes:**

| Outcome | Description |
| --- | --- |
| ACK | Commitment request accepted for processing. |
| NACK | Rejected — invalid payload. (Eligibility failures — unknown block, wrong status, non-sequential round — are workflow outcomes, not admission failures.) |

### CC_CHECK_CHAIN_NOT_INITIALIZED_V0
**Summary:** Enforces the single-genesis invariant by refusing initialization when a chain head already exists.
**Inputs:**
_No inputs declared (reads the fixed chain state record)._
**Outputs:**
_No outputs declared._
**Result Statuses:** `SUCCESS`, `ALREADY_EXISTS`, `VIOLATION`, `BACKEND_ERROR`

| Step | Purpose |
