## Design Pivot Record

**Decision date:** 2026-06-03
**Trigger:** Analyst architectural review prior to Stage 6b approval

**Decision:** Replace the unified `WF_SUBMIT_TRANSACTION_V0` with **eight typed transaction workflows** — one per transaction type.

**Rationale (four architectural arguments + one governance argument):**

| # | Argument |
|---|----------|
| a | Asymmetric schema — MINT has no from_address, SLASH has no to_address. A single IN_ cannot govern these without conditional nullability, which is schema ambiguity |
| b | Different processing semantics — TRANSFER needs balance + nonce; MINT needs supply gate; REWARD needs POOL balance; SLASH needs penalty computation. A single CC chain cannot declare these as compile-time paths |
| c | Security enforcement at WF/IN_ boundary — MINT, POOL, REWARD, SLASH, BURN are SYSTEM-authority operations. A unified ENDUSER IN_ cannot enforce SYSTEM-only access. Separate WFs bind each type to its declared authority class |
| d | Extensibility — new transaction types (CONTRACT_CALL, GOVERNANCE) are added as new WFs with zero impact on existing WF topology |
| e | **PGS compile-time resolution (doctrine)** — `CC_VALIDATE_TX_POLICY_V0` branching on tx_type at runtime is an undeclared conditional graph. The compiler cannot see or validate it. Separate WFs make the full execution path explicit in the governed topology at compile time |

**tx_type defect root cause (now understood):** The hardcoded `tx_type: "ETH"` in two WF node bindings (`CC_VALIDATE_TX_POLICY_V0` and `CC_PERSIST_MEMPOOL_TX_V0`) is a symptom of the architectural defect, not the primary defect. The primary defect is that a typed concept is carried as a payload field through a topology that has no type-aware structure. The fix is typed WFs — not a binding patch.

**Scope impact of this pivot:**
- 8 new WFs, 8 new IN_, 8 new RB_, 8 new type-specific CC_ (policy gates)
- 6 shared CC_ (some new, some reuse/update)
- 8 artifacts retired (WF, IN, RB from unified path; 5 CCs superseded)
- Data model IN_ updates (Actor, Wallet, Validator), CC updates (data model enrichment), events, and seeds are UNCHANGED from Stage 6b as originally drafted

**Change management learning objective:** This pivot exercises governed WF retirement — the first instance of retiring an existing live artifact in this CR pipeline. Section 2 below documents the retirement pattern.

## 1. Design Decisions Resolution

| Decision | Design Resolution |
|----------|-------------------|
| Genesis actor authority model | **GENESIS STATE BYPASS** — genesis_actor and system wallets are pre-committed seed state loaded at bootstrap. The GENESIS actor bypasses all workflow participation (no IN_, WF_, CC_). Its authority is a bootstrap fact, not a runtime claim. The 1,000,000 BACHI in MINT is committed genesis state — not produced by WF_MINT_V0. This breaks the circular dependency: the block pipeline depends on a funded state to function; genesis state must predate it. All subsequent token movements are normal governed operations. |
| Unified vs typed transaction WFs | **TYPED** — 8 WFs, one per transaction type. No runtime type branching in CC chains |
| Authority split for tx types | ENDUSER: TRANSFER, STAKE, UNSTAKE. SYSTEM: MINT, BURN, POOL, REWARD, SLASH |
| CC_BUILD_ETH_TX_V0, CC_SIGN_TRANSACTION_V0, CC_HASH_TRANSACTION_V0 | **RETIRED** — KISS removes signing. tx_hash generated by CT_PURE_GENERATE_ID_V0 |
| CC_VALIDATE_TX_STRUCTURE_V0 | **RETIRED** — structural validation absorbed into typed IN_ admission gates |
| CC_VALIDATE_TX_POLICY_V0 | **RETIRED** — replaced by 8 type-specific CC_VALIDATE_<TYPE>_POLICY_V0 |
| Balance check CC — shared or inline | **INLINE** in type-specific policy CC — keeps topology lean; KISS single-node |
| Nonce reservation for SYSTEM types | **NULL** — MINT, BURN, POOL, REWARD, SLASH carry nonce=null (system-generated, no replay risk) |
| Actor resolution for SYSTEM types | **OMITTED** — SYSTEM WFs carry triggered_by (audit only); no actor_record required |
| tx_hash generation post-signing removal | `CT_PURE_GENERATE_ID_V0` (prefix: TX, hex output) — deterministic, replay-safe |
| WF topology changes for data model enrichments | **NONE** — Actor, Wallet, Validator, Block WF topologies unchanged; changes are CC-level only |
| Actor status at registration | `status = PENDING` — CC-injected literal constant |
| Wallet EOA address | KISS: `"0x" + wallet_id.replace("-","")[:40]` — no crypto derivation this CR |
| Wallet nonce at creation | `nonce = 0` — literal constant |
| validator_index assignment | count+1 from VALIDATORS store at write time |
| epoch for blocks | `height // 32` — computed in CC_FORM_BLOCK_V0 assembly step |
| block_type | `"GENESIS" if height == 0 else "REGULAR"` — conditional constant in assembly step |
| base_fee_per_gas for blocks | KISS constant: `1000000000` wei |
| New CT artifacts | **NONE** — all computations reuse CT_PURE_ASSEMBLE_RECORD_V0 + CT_PURE_GENERATE_ID_V0 + literal injection |

## 2. Retirement Register

**Governed artifact retirement** — first instance in this CR pipeline. A retired artifact carries `status: RETIRED` and `superseded_by` in its protocol source. The runtime refuses to execute a RETIRED workflow. The compiler flags any artifact still referencing a RETIRED artifact as a dependency error.

| Artifact | FQDN | Retired By | Reason |
|----------|------|------------|--------|
| `WF_SUBMIT_TRANSACTION_V0` | `blockchain::WF_SUBMIT_TRANSACTION_V0` | 8 typed WFs | Unified tx WF superseded by typed WF per transaction class |
| `IN_TRANSACTION_SUBMITTED_V0` | `blockchain::IN_TRANSACTION_SUBMITTED_V0` | 8 typed IN_ | Single unified IN_ replaced by typed admission per tx class |
| `RB_SUBMIT_TRANSACTION_V0` | `blockchain::RB_SUBMIT_TRANSACTION_V0` | 8 typed RB_ | Runtime binding for retired WF |
| `CC_VALIDATE_TX_STRUCTURE_V0` | `blockchain::CC_VALIDATE_TX_STRUCTURE_V0` | Typed IN_ gates | Structural validation absorbed into typed IN_ admission |
| `CC_VALIDATE_TX_POLICY_V0` | `blockchain::CC_VALIDATE_TX_POLICY_V0` | 8 typed policy CCs | Runtime tx_type branching replaced by per-type CC |
| `CC_BUILD_ETH_TX_V0` | `blockchain::CC_BUILD_ETH_TX_V0` | CT_PURE_GENERATE_ID_V0 | ETH unsigned tx builder; no signing in KISS |
| `CC_SIGN_TRANSACTION_V0` | `blockchain::CC_SIGN_TRANSACTION_V0` | — | ECDSA signing removed in KISS demo |
| `CC_HASH_TRANSACTION_V0` | `blockchain::CC_HASH_TRANSACTION_V0` | CT_PURE_GENERATE_ID_V0 | Keccak-256 hash replaced by deterministic ID generation |

**Retirement authoring obligation:** Each retired artifact must have its protocol source updated with `status: RETIRED` and `superseded_by: [list of successor FQDNs]` before the new typed artifacts are compiled. Build order: retire first, then compile successors.

## 6. Execution Topologies — 8 Typed WFs

### ENDUSER topology pattern (TRANSFER, STAKE, UNSTAKE)

```
IN_<TYPE>_V0
    ACK  → CC_RESOLVE_ACTOR_ID_V0
    NACK → EXIT

CC_RESOLVE_ACTOR_ID_V0
    SUCCESS       → CC_VALIDATE_<TYPE>_POLICY_V0
    NOT_FOUND     → EXIT
    VIOLATION     → EXIT
    BACKEND_ERROR → EXIT

CC_VALIDATE_<TYPE>_POLICY_V0          ← type-specific: checks wallet ownership,
    SUCCESS       → CC_RESERVE_NONCE_V0   balance, address rules; returns from_address
    NOT_FOUND     → EXIT                  and to_address resolved
    VIOLATION     → EXIT
    BACKEND_ERROR → EXIT

CC_RESERVE_NONCE_V0
    SUCCESS       → CC_PERSIST_MEMPOOL_TX_V0
    VIOLATION     → EXIT
    BACKEND_ERROR → EXIT

CC_PERSIST_MEMPOOL_TX_V0
    SUCCESS          → CC_APPEND_TX_EVENT_V0
    ALREADY_EXISTS   → EXIT
    VIOLATION        → EXIT
    BACKEND_ERROR    → EXIT

CC_APPEND_TX_EVENT_V0
    SUCCESS       → EXIT
    VIOLATION     → EXIT
    BACKEND_ERROR → EXIT

EXIT
```

### SYSTEM topology pattern (MINT, POOL, REWARD)

```
IN_<TYPE>_V0                          ← SYSTEM authority; no actor_record
    ACK  → CC_VALIDATE_<TYPE>_POLICY_V0
    NACK → EXIT

CC_VALIDATE_<TYPE>_POLICY_V0          ← resolves from/to addresses; checks
    SUCCESS       → CC_PERSIST_MEMPOOL_TX_V0   system wallet balances
    NOT_FOUND     → EXIT
    VIOLATION     → EXIT
    BACKEND_ERROR → EXIT

CC_PERSIST_MEMPOOL_TX_V0              ← nonce=null for system txs
    SUCCESS          → CC_APPEND_TX_EVENT_V0
    ALREADY_EXISTS   → EXIT
    VIOLATION        → EXIT
    BACKEND_ERROR    → EXIT

CC_APPEND_TX_EVENT_V0
    SUCCESS       → EXIT
    VIOLATION     → EXIT
    BACKEND_ERROR → EXIT

EXIT
```

### BURN / SLASH topology (SYSTEM authority + source wallet balance check)

Same as SYSTEM pattern above. Balance check is inline in `CC_VALIDATE_BURN_POLICY_V0` / `CC_VALIDATE_SLASH_POLICY_V0` — source wallet must have sufficient balance before routing to BURN sink.

## 7. Type-Specific Policy CC — Design Contracts

Each policy CC handles the full type-specific semantics: wallet resolution, balance gate, address routing, output of resolved from_address and to_address for the persist step.

| CC | Wallet checks | Balance gate | Address routing | Output |
|----|---------------|-------------|-----------------|--------|
| CC_VALIDATE_TRANSFER_POLICY_V0 | from_wallet belongs to actor | from_wallet.balance ≥ amount | from=from_wallet.address; to=payload.to_address | from_address, to_address |
| CC_VALIDATE_STAKE_POLICY_V0 | from_wallet belongs to actor | from_wallet.balance ≥ amount | from=from_wallet.address; to=POOL wallet address (auto) | from_address, to_address |
| CC_VALIDATE_UNSTAKE_POLICY_V0 | validator_index belongs to actor | POOL.staked_for_validator ≥ amount | from=POOL wallet address (auto); to=to_wallet.address | from_address, to_address |
| CC_VALIDATE_MINT_POLICY_V0 | MINT wallet exists | MINT.balance ≥ amount | from=MINT wallet address (auto); to=to_wallet.address | from_address, to_address |
| CC_VALIDATE_BURN_POLICY_V0 | from_wallet exists | from_wallet.balance ≥ amount | from=from_wallet.address; to=BURN wallet address (auto) | from_address, to_address |
| CC_VALIDATE_POOL_POLICY_V0 | MINT + POOL wallets exist | MINT.balance ≥ amount | from=MINT wallet address (auto); to=POOL wallet address (auto) | from_address, to_address |
| CC_VALIDATE_REWARD_POLICY_V0 | POOL wallet exists, to_wallet exists | POOL.balance ≥ amount | from=POOL wallet address (auto); to=to_wallet.address | from_address, to_address |
| CC_VALIDATE_SLASH_POLICY_V0 | from_wallet (validator) exists | from_wallet.balance ≥ amount | from=from_wallet.address; to=BURN wallet address (auto) | from_address, to_address |

**SYSTEM wallet resolution:** MINT, BURN, POOL wallet addresses are resolved by wallet_type lookup from seeds/system_wallets.json — not hardcoded. CC reads WALLETS store filtered by `wallet_type = MINT/BURN/POOL`.

## 8. Shared CC Pipeline — Updated CC_PERSIST_MEMPOOL_TX_V0

**CC_PERSIST_MEMPOOL_TX_V0 (UPDATE)**

This CC is shared across all 8 typed WFs. `tx_type` is now passed as a literal constant from each WF node binding — the WF topology is the type, eliminating the need for a tx_type payload field.

- Step 1: `CT_PURE_GENERATE_ID_V0` (prefix: TX, hex output) → tx_hash
- Step 2: `CT_PURE_ASSEMBLE_RECORD_V0` — assemble: `{tx_hash, tx_type (WF-literal), from_address (from policy CC result), to_address (from policy CC result), amount, currency: "BACHI", network: "TESTNET", nonce (from CC_RESERVE_NONCE_V0 or null for SYSTEM), gas_limit, max_fee_per_gas, max_priority_fee_per_gas, base_fee_per_gas: null, effective_gas_price: null, gas_used: null, total_fee: null, block_number: null, block_hash: null, status: "PENDING", memo, created_at (JIT)}`
- Step 3: `CS_APPENDONLY_JSONL_V0` — APPEND — MEMPOOL store → persist transaction
- Result statuses: `SUCCESS`, `ALREADY_EXISTS`, `VIOLATION`, `BACKEND_ERROR`
- Output: `tx_hash`

*Execution result fields (base_fee_per_gas, effective_gas_price, gas_used, total_fee, block_number, block_hash) are null at submission — populated by consensus finalization workflows in next CR.*

## 9. Data Model CC Pipelines (Unchanged from original Stage 6b)

### CC_WRITE_ACTOR_RECORD_V0 (UPDATE)
- Step 1: `CT_PURE_ASSEMBLE_RECORD_V0` — assemble with status: "PENDING" (literal), all enriched fields, created_at/last_modified (JIT)
- Step 2: `CS_MUTABLE_JSON_V0` — WRITE — ACTORS store (key: actor_id)
- Output: actor_id

### CC_CREATE_WALLET_RECORD_V0 (UPDATE)
- Step 1: `CT_PURE_GENERATE_ID_V0` (prefix: WAL) → wallet_id
- Step 2: `CT_PURE_ASSEMBLE_RECORD_V0` — assemble with status: "ACTIVE" (literal), nonce: 0, address: `"0x" + wallet_id.replace("-","")[:40]`, last_tx_at: null
- Step 3: `CS_MUTABLE_JSON_V0` — WRITE — WALLETS store (key: wallet_id)

### CC_WRITE_VALIDATOR_RECORD_V0 (UPDATE)
- Step 1: `CS_MUTABLE_JSON_V0` — LIST — VALIDATORS store → count
- Step 2: `CT_PURE_ASSEMBLE_RECORD_V0` — assemble with validator_index = count+1, all enriched fields
- Step 3: `CS_MUTABLE_JSON_V0` — WRITE — VALIDATORS store (key: validator_index)

### CC_FORM_BLOCK_V0 (UPDATE)
- Step 1: `CT_PURE_GENERATE_ID_V0` (prefix: BLK, hex) → block_hash
- Step 2: `CT_PURE_ASSEMBLE_RECORD_V0` — compute epoch = height//32, block_type by height==0, base_fee_per_gas constant; assemble full enriched entity with status: "PROPOSED"
- Step 3: `CS_MUTABLE_JSON_V0` — WRITE — BLOCKS store
- Step 4: `CS_APPENDONLY_JSONL_V0` — APPEND — BLOCK_EVENTS (EV_BLOCK_PROPOSED_V0)

## 12. Governance Decision Gate

**Presenting for Analyst approval:**

0. **D0 — Genesis State Bypass** — The GENESIS actor is a special system actor with elevated authority that exists outside the governed workflow pipeline. The GENESIS actor and system wallets (MINT 1,000,000 BACHI, BURN 0, POOL 0) are pre-committed as seed state, loaded at bootstrap. This bypasses actor registration, wallet creation, and the entire transaction/block/finalization pipeline. Architectural necessity: the block pipeline requires a funded state to function; genesis state must predate and bootstrap the pipeline, not be produced by it. The GENESIS actor has no IN_, WF_, or CC_ — its authority is a bootstrap fact. WF_MINT_V0 governs subsequent minting operations post-bootstrap; it does not produce the initial 1M supply.
1. **Design pivot approved** — typed WF per tx type replaces unified WF; rationale documented in pivot record above
2. **8 artifacts retired** — WF, IN, RB from unified path; 5 CCs superseded; retirement authoring is required before successor compilation
3. **Authority split** — TRANSFER/STAKE/UNSTAKE = ENDUSER authority; MINT/BURN/POOL/REWARD/SLASH = SYSTEM authority; enforced at IN_ level per WF
4. **tx_type elimination from payload** — tx_type is no longer a payload field in any IN_; it is a WF-literal injected into CC_PERSIST_MEMPOOL_TX_V0 node binding; payload ambiguity eliminated
5. **SYSTEM wallet auto-resolution** — MINT/BURN/POOL addresses resolved by wallet_type lookup from WALLETS store; not hardcoded in any CC
6. **Balance check inline in policy CC** — not a separate shared CC; KISS single-node; type-specific semantics warrant inline
7. **Nonce = null for SYSTEM types** — MINT/BURN/POOL/REWARD/SLASH skip CC_RESERVE_NONCE_V0; system txs have no replay risk in single-node
8. **Signing pipeline retired** — CC_BUILD_ETH_TX_V0, CC_SIGN_TRANSACTION_V0, CC_HASH_TRANSACTION_V0 removed; tx_hash from CT_PURE_GENERATE_ID_V0
9. **Structural validation absorbed into IN_** — CC_VALIDATE_TX_STRUCTURE_V0 retired; typed IN_ admission gates enforce field types, required fields, and format constraints per type
10. **No new CT, CS, or authority class** — all new behavior expressed in typed WF topology + typed CC policy gates
11. **Data model enrichments unchanged** — Actor/Wallet/Validator/Block CC updates carry forward exactly as originally drafted

### Amendment A — CC_RESERVE_NONCE_V0 not used

**DI location:** Section 3 (Shared CCs table) — `CC_RESERVE_NONCE_V0` row: "REUSE — Sound — used by ENDUSER typed WFs; SYSTEM WFs skip"

**DI location:** Section 11 (Artifact Summary) — REUSE row lists `CC_RESERVE_NONCE_V0`

**As-built:** `CC_RESERVE_NONCE_V0` is absent from all 8 typed transaction WF pipelines (ENDUSER and SYSTEM).

**Why:** The BACHI data model uses content-addressable transaction identity (`tx_id` derived from payload hash via `CT_PURE_GENERATE_ID_V0`). There is no sequential nonce counter in a content-addressable ID system. Nonce reservation was an ETH-style mechanism inherited from the retired `WF_SUBMIT_TRANSACTION_V0`; it has no role in the BACHI-native pipeline.

**ENDUSER WF corrected topology:**
```
CC_VALIDATE_<TYPE>_POLICY_V0
    SUCCESS → CC_GENERATE_TX_ID_V0
CC_GENERATE_TX_ID_V0
    SUCCESS → CC_PERSIST_MEMPOOL_TX_V0
```
(See Amendment C for `CC_GENERATE_TX_ID_V0`.)

### Amendment B — CC_WRITE_ACTOR_RECORD_V0 not used in transaction WFs

### Amendment C — CC_GENERATE_TX_ID_V0 (unplanned addition)

**DI location:** Absent from this document entirely — not planned, not listed in any section.

**As-built:** `CC_GENERATE_TX_ID_V0` added as an unplanned artifact during Wave 4 authoring.

**Why:** Retiring `CC_BUILD_ETH_TX_V0` and `CC_HASH_TRANSACTION_V0` (ETH pipeline) left no mechanism for generating `tx_id` and `tx_hash` in the BACHI-native pipeline. Content-addressable; no cryptographic signing. The total mandated authoring count moved from 57 to 58.

### Amendment D — tx_type injection mechanism changed

**DI location:** Section 8 (CC_PERSIST_MEMPOOL_TX_V0 update specification) — describes `tx_type` injected via "WF context literal" through a `wf_literal` binding declared in the RB_ artifact.

**As-built:** `wf_literal` RB binding pattern was not implemented in the compiler. `tx_type` is declared as a YAML literal value directly in each WF's `inputs` block for the `CC_GENERATE_TX_ID_V0` and `CC_PERSIST_MEMPOOL_TX_V0` nodes.

**Why:** No `wf_literal` mechanism was found in the compiler at authoring time. The YAML literal pattern is semantically equivalent — each typed WF is self-contained and carries its own `tx_type` constant. `wf_literal` is deferred to a future CR as a potential compiler feature candidate.

inputs:
